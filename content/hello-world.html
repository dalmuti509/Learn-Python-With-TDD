Your First Python Program with TDD

Let's start with the classic "Hello, world" program, but we'll do it the TDD way!

Step 1: Write Your First Test (Red Phase)

Start by writing a test that describes what you want your function to do. This test will fail initially because the function doesn't exist yet.

ðŸ’¡ Understanding the Test

What we're testing: We want a function called hello() that returns the text "Hello, world"

Key concepts:

  • Function definition: def test_hello_returns_hello_world(): - This creates a test function
  • Function call: result = hello() - This calls the function we want to create
  • Assertion: assert result == "Hello, world" - This checks if the result matches our expectation
  • String comparison: We're comparing the returned string with the exact text "Hello, world"
hello_world_test.py
def test_hello_returns_hello_world(): result = hello() assert result == "Hello, world"

Run the test: pytest hello_world_test.py -v

Expected result: The test fails because the hello() function doesn't exist yet.

Step 2: Write Minimal Implementation (Green Phase)

Now write the minimal code to make the test pass. Don't over-engineer it!

ðŸ’¡ Understanding the Implementation

What we need to create: A function called hello() that returns "Hello, world"

Key concepts for the implementation:

  • Function definition: def hello(): - This creates a function named "hello"
  • Return statement: return "Hello, world" - This sends the text back to whoever calls the function
  • String literal: "Hello, world" - This is the exact text we want to return (note the quotes)
  • Minimal approach: We're writing the simplest possible code that makes the test pass

Why this works: When the test calls hello(), our function returns "Hello, world", which matches exactly what the test expects.

hello_world.py
def hello(): return "Hello, world"

Run the test again: pytest hello_world_test.py -v

Expected result: The test now passes! ðŸŽ‰

Step 3: Refactor (Optional)

Since this is a simple function, there's not much to refactor. But in more complex code, this is where you'd improve the code while keeping the tests green.
